🔴 Críticas — Ação Imediata
Corrigir erro "cannot read properties of undefined (reading 'autoplay')"
Ocorreu em todos os 8 dias com 200 sessões afetadas (51,2% de todos os erros JS). Provavelmente causado por um plugin de slider/carrossel (Swiper, Owl Carousel ou similar) que tenta inicializar o autoplay antes do elemento existir no DOM. Verificar ordem de carregamento dos scripts e adicionar verificação de existência do elemento antes de chamar o método autoplay.
Componente: plugin de slider/carrossel | Afeta: todos os dispositivos | Impacto: UX degradada para ~25 sessões/dia
Resolver "wp is not defined" e "can't find variable: wp"
Juntos somam 96 sessões afetadas no período. O objeto global wp do WordPress está sendo referenciado antes de ser carregado, ou está sendo removido por algum plugin de otimização (ex: WP Rocket, Autoptimize). Verificar se o script wp-embed.min.js ou wp-api.js está sendo deferido/adiado incorretamente. Garantir que dependências do WordPress sejam carregadas antes dos scripts customizados.
Componente: WordPress Core / Plugin de otimização | Afeta: todos os dias | Impacto: funcionalidades WordPress quebradas
Investigar pico de bots no dia 18/04 — 76,6% do tráfego
Em 18/04, 298 de 389 sessões foram bots. Isso sugere um ataque de scraping, varredura de segurança ou tráfego inválido. Verificar logs do servidor para aquele dia, identificar os IPs de origem e considerar implementar proteção adicional (Cloudflare Bot Management, reCAPTCHA, rate limiting). Os dados comportamentais do 18/04 devem ser interpretados com cautela.
Data: 18/04/2026 | Impacto: dados distorcidos, possível sobrecarga do servidor, custo de infraestrutura
🟡 Alta Prioridade
Melhorar LCP (Largest Contentful Paint) — média 3,3s acima do alvo de 2,5s
O LCP ficou acima de 2,5s em todos os 8 dias (máximo: 3,712s em 20/04). As principais causas prováveis são: imagens hero sem lazy loading, fonts externas bloqueando renderização, e CSS não crítico inline. Ações: otimizar imagens acima da dobra com WebP + preload, adicionar font-display: swap, e usar Critical CSS para o above-the-fold.
Afeta: taxa de conversão e Core Web Vitals (Google Ranking) | Melhor dia: 18/04 (2,695s) — analisar o que foi diferente
Corrigir CLS crítico nos dias 17/04 (0,515) e 18/04 (0,510)
O CLS (Cumulative Layout Shift) esteve acima de 0,5 em dois dias — mais de 5x o limite aceitável de 0,1. Esses saltos de layout afetam diretamente a experiência do usuário, especialmente em mobile. Verificar se anúncios sem dimensões fixas, fontes web causando FOUT ou imagens sem width/height declarados estão contribuindo. A página g4-learning mostrou CLS de 1,183 (14/04), 1,419 (18/04) e 1,147 (19/04) — prioridade máxima para essa URL.
Página crítica: g4business.com/g4-learning | Afeta: Core Web Vitals e ranking Google
Reduzir taxa de Dead Clicks (cliques inativos) — média de ~19% das sessões
Em média, 19% das sessões têm pelo menos um clique inativo (clique em elemento não clicável). Picos: 20/04 com 23,51%. Isso indica que usuários estão tentando clicar em imagens, textos ou elementos que parecem interativos mas não são. Usar os mapas de calor do Clarity para identificar exatamente quais elementos estão recebendo esses cliques e ajustar o design (adicionar cursor pointer onde necessário, tornar elementos clicáveis ou remover a aparência interativa).
Afeta: UX e taxa de conversão | Ação: revisar heatmaps de clique no Clarity
Investigar alta taxa de clique com botão direito (~20% das sessões)
A taxa de botão direito oscilou entre 16,5% e 23,1% das sessões — incomumente alta. Pode indicar: (a) usuários tentando copiar conteúdo (textos de blog, imagens de mentores), (b) tentativa de abrir links em nova aba, ou (c) comportamento de scraping humano. Verificar no Clarity quais elementos recebem mais clique direito e avaliar se é necessário proteger conteúdo ou otimizar CTAs para "abrir em nova aba".
Afeta: propriedade intelectual do conteúdo e análise de intenção do usuário
Otimizar INP nos dias com 360ms — páginas lentas para interação
O INP (Interaction to Next Paint) atingiu 360ms em 14/04 e 20/04, acima do limite de 200ms. A homepage (g4business.com/) aparece repetidamente com INP alto (504ms em 20/04). Causas: JavaScript pesado bloqueando a thread principal, event listeners ineficientes. Ação: auditar com Chrome DevTools > Performance, identificar long tasks e quebrar em chunks menores com scheduler ou requestIdleCallback.
Afeta: Core Web Vitals desde 2024 | Página principal afetada: g4business.com/
🔵 Média Prioridade
Corrigir "script error." — erros cross-origin sem detalhes rastreáveis
55 ocorrências no período. "Script error." aparece quando um script externo de outro domínio lança uma exceção sem o header CORS Access-Control-Allow-Origin. Verificar scripts de terceiros (GTM, HubSpot, pixels de redes sociais) e garantir que têm o atributo crossorigin="anonymous" e que os servidores respondem com os headers corretos.
Origem provável: scripts de marketing/analytics de terceiros | Afeta: rastreabilidade de erros
Investigar e revisar páginas inativas com alto potencial de tráfego
As páginas inativas se repetem nos 8 dias — especialmente posts de blog (deepseek, elon-musk-vs-chatgpt, papa-francisco, planejamento-tributario) e glossário. Estas páginas existem no site mas não recebem tráfego orgânico, possivelmente por problemas de indexação, links quebrados ou conteúdo desatualizado. Fazer auditoria de SEO técnico (Google Search Console) e atualizar os posts de maior potencial.
Páginas afetadas: /blog/* e /glossario/* | Impacto: perda de tráfego orgânico potencial
Monitorar e otimizar tráfego mobile — picos de 72% (19/04)
O tráfego mobile variou de 34% a 72% dependendo do dia, com picos fortes aos fins de semana. Garantir que todas as páginas principais estejam 100% responsivas, com botões de CTA com tamanho mínimo de 44px de toque, e que o LCP mobile seja monitorado separadamente (tende a ser pior que desktop).
Dias com dominância mobile: 18/04 (63%), 19/04 (72%), 21/04 (53%)
Configurar alerta de bots e implementar filtro no Clarity
O volume de bots variou de 3% a 77% do tráfego. Implementar: (1) filtro de segmento no Clarity excluindo sessões de bot para análise comportamental limpa; (2) alerta no servidor ou Cloudflare para picos de tráfego automatizado acima de 30%; (3) revisar robots.txt e considerar honeypot para identificar bots maliciosos.
Dia crítico: 18/04 com 298 bots | Afeta: qualidade dos dados de análise
